# SoulGo MVP 调整规划文档

基于您提出的四大核心要点，针对当前 `prototype.html` 的 MVP 版本，制定以下调整规划。目标是将产品从“功能型原型”通过架构调整和体验优化，转向“灵魂孵化器”的平台型产品雏形。

## 1. 核心理念调整：人格第一，功能第二

**现状分析**：
目前的 MVP 侧重于“打卡”功能和简单的状态机（待机、移动、交互），人格主要通过 CSS 颜色和简单的文案差异体现，略显单薄。

**调整方案 - 落地功能点**：
*   **“灵魂”具象化**：
    *   **移除/弱化非核心 UI**：将功能性按钮（如设置、地图切换等）收纳或弱化，让屏幕的 C 位完全留给“灵魂”（角色）。
    *   **动态待机 (Live Idle)**：角色不再静止。它会随机执行“微动作”：眨眼、伸懒腰、打瞌睡、或是盯着屏幕前的你（视线跟随）。使用 CSS 动画切换图片帧配合随机时间间隔逻辑。
    *   **情绪气泡 (Mood Bubbles)**：角色头顶会不定时冒出“想法气泡”。例如小火苗：“好想出去跑一圈啊！”（配合火焰跳动特效）；小石头：“...（发呆中）...”（配合静止省略号）。
*   **对话即交互**：
    *   不仅是“点击触发动画”，而是建立“对话感”。用户的每一次操作（点击、抚摸、喂食），角色都应有基于其当前情绪和性格的反馈（不仅仅是动作，配合气泡文字或表情符号）。
*   **成长性设计**：
    *   引入简单的“亲密度/灵性”等级。随着交互次数增加，角色的回应会发生变化（例如：从陌生到依赖）。

## 2. 交互逻辑重构：情感驱动，去任务化

**现状分析**：
当前包含“打卡”、“收集”等功能，容易给用户一种“为了收集而打卡”的任务感。

**调整方案 - 落地功能点**：
*   **“打卡”重构为“共游/记忆”**：
    *   **共游日记 (Travel Memories)**：不再强调“打卡地点”，而是强调“我们一起去了哪里”。用户输入地点，角色“出门”，一段时间后带回**“拍立得照片”**（AI 生成的地点图）和一篇**“日记”**（以角色口吻描述经历）。
    *   **照片墙 (Memory Wall)**：生成的拍立得照片直接贴在房间背景的墙上，随着时间推移成为独一无二的回忆角落。
*   **“喂养”重构为“能量传递”**：
    *   **能量传递 (Energy Transfer)**：替代传统的“喂食按钮”。用户长按屏幕下方的“能量指纹区”，粒子流从指尖飞向角色，角色身体逐渐变亮/变大，表示充能完毕。
*   **家园即心流空间**：
    *   家园不仅是背景，而是角色“存放回忆”的地方。
    *   用户带回来的“特产/照片”会随机出现在家园中，角色会与之互动（比如看着照片发呆），触发回忆对话。

## 3. 软硬数据双向流动（MVP 模拟方案）

**现状分析**：
目前完全是纯软件交互，暂时不考虑硬件联动。

**调整方案 - 落地功能点**：
*   **模拟生命体征（输出）**：
    *   **心跳共鸣 (Heartbeat Sync)**：UI 界面（如背景的光影）会以特定频率律动，模拟角色的“心跳”。激动时律动快且暖色，平静时律动慢且冷色。
*   **模拟触摸反馈（输入）**：
    *   **抚摸反馈 (Touch & Stroke)**：区分“轻点 (Poke)”（角色惊讶/打招呼）和“长按 (Stroke/Hug)”（角色闭眼享受，屏幕边缘泛光，手机微震）。

## 4. 架构重构：平台化预留 (IP 中立的情感引擎)

**现状分析**：
当前代码逻辑（游戏循环、渲染、数据）耦合在单个 HTML 文件中，且与具体 IP（小火苗等）绑定较紧。

**调整方案 - 落地功能点**：
*   **抽象“灵魂引擎” (SoulEngine)**：
    *   将核心逻辑抽离：`EmotionSystem` (情绪计算), `InteractionHandler` (交互处理), `GrowthSystem` (成长逻辑)。
    *   这部分代码不包含任何具体 IP 的图片或文案。
*   **配置化 IP (SoulProfile)**：
    *   **灵魂切换 (Soul Slot)**：建立标准的 JSON 配置文件，定义 IP 的 Assets、Personality、Behaviors。通过加载不同的 JSON 配置，可以在同一个网页中无缝切换完全不同的两个角色（例如从“小火苗”切换到“小石头”），验证引擎的通用性。
*   **标准化协议 (SoulProtocol)**：
    *   定义前端与“硬件”的通信协议，当前阶段仅在软件内部模拟信号。

## 下一步执行计划 (MVP 修改)

1.  **代码结构拆分**：将 `prototype.html` 拆分为 `engine.js` (核心逻辑), `data.js` (IP配置), `view.js` (渲染), `main.js` (入口)。
2.  **数据结构定义**：定义标准的 IP 配置文件格式 (JSON)。
3.  **交互优化**：实现“长按抚摸”与“模拟心跳”的视觉效果。
4.  **去任务化 UI**：简化界面，突出角色与情感反馈。
